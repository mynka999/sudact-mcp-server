# Доказательная статистика по судебной практике (sudact)

Скилл повторяет метод из статьи А.Зазулина (sudact + парсер + LLM), но
исправляет его методологические дыры. Бэкенд сбора — `stat_corpus.py` в
репозитории sudact-mcp; кодирование и агрегацию делает модель по этому скиллу.

**Путь к бэкенду:** `C:\Users\USER\Documents\Veritas\Code\sudact-mcp\stat_corpus.py`
**Запуск:** `py -X utf8 <путь> ...` (зависимость — пакет `mcp`, уже стоит).

## Жёсткие принципы (не нарушать)

1. **Никогда не называть результат «достоверной/генеральной статистикой».**
   Это всегда *оценочная статистика по доступной выборке*. sudact отдаёт максимум
   ~500 результатов на запрос (пагинация зацикливается на 50-й странице) и только
   опубликованную, обезличенную часть актов.
2. **Не путать точность извлечения с репрезентативностью.** Можно идеально
   закодировать смещённую выборку — выводы всё равно будут смещены.
3. **Честная выборка обязательна.** Никаких «первых 100». Только случайная
   выборка (`random`) или, для больших корпусов, стратификация по датам
   (`stratified`), которая покрывает весь период, а не верхушку сортировки.
4. **Каждая итоговая цифра должна опираться на конкретные решения** со ссылками
   на sudact (из собранного corpus.jsonl). Считать «не определено» отдельной
   категорией, а не молча выкидывать.

## Процедура

### Шаг 1. Уточнить задачу
Сформулировать с пользователем:
- исследовательские вопросы (например: % реальных сроков; смежные составы;
  медиана взысканной суммы);
- поисковые параметры sudact: `--query` (короткие ключевые слова!),
  опц. `--article`, `--scope` (regular|arbitral|vsrf|magistrate);
- период анализа и желаемый размер выборки `--n` (по умолчанию 30–50;
  больше = точнее, но дороже по токенам на кодировании).

### Шаг 2. Разведка объёма
Узнать `total_found` (одним вызовом `search_court_practice` или
`stat_corpus.py --mode random --n 1`). Решить режим:
- **total ≲ 500** → `--mode random` достанет почти весь корпус.
- **total большой** → `--mode stratified --windows ...` по годам. Для лет, где
  по году всё ещё >500 (видно в отчёте: `reachable=500 < total_found`), дробить
  год на кварталы/месяцы, чтобы каждое окно было достижимо целиком.

### Шаг 3. Собрать выборку (с текстами)
```
py -X utf8 stat_corpus.py --query "<...>" [--article "<...>"] --scope regular \
   --mode stratified --windows "2022,2023,2024,2025" --n 50 --with-text \
   --seed 42 --out corpus.jsonl
```
Сохранить из вывода отчёт о методике (total_found_overall, reachable_overall,
разбивка по окнам, picked). Он пойдёт в справку дословно.

### Шаг 4. Кодирование решений
Прочитать `corpus.jsonl` (поле `text_excerpt` — фабула + резолютивная часть).
По каждому решению извлечь поля под вопросы в строгую схему, например для
уголовного кейса:
`{doc_id, url, основной_состав, смежные_составы[], вид_наказания, срок,
реальное_или_условное, сумма_взыскания|null, прочее, не_определено?}`.

**Модель кодировщика: Sonnet** (`agent(prompt, {model:'sonnet', schema})`).
Калибровка 2026-06-26 (20 дел): Sonnet vs Opus совпали 20/20 по ИСХОДУ и
РЕЛЕВАНТНОСТИ (критичные поля), 18/20 по статье и 19/20 по марке — и оба
расхождения по статье были на пограничных делах, где Sonnet не хуже. Opus здесь
не нужен; Haiku не проверялся и для статистики рискован.

**Кормить агенту ОБРЕЗКУ** (`text_excerpt` из `trim_operative`), а НЕ полный
текст: вход ~×11 дешевле без потери точности (полный текст съедает ~46k токенов
на дело). Обрезка = голова + блок от последнего «ПРИГОВОРИЛ/ПОСТАНОВИЛ/РЕШИЛ».

**`основной_состав` при совокупности неоднозначен** — фиксируй НАИБОЛЕЕ ТЯЖКУЮ
вменённую статью, а все прочие клади в `смежные_составы`. Иначе модели расходятся
(одна берёт ч.1, другая п.г ч.3 того же 158).

- Для выборки ≤ ~40 — кодировать в контексте.
- Для большой выборки — многоагентный **Workflow** (fan-out по решениям) с явного
  согласия пользователя (жжёт токены). В воркфлоу: `.catch(()=>null)` на каждом
  агенте + фильтр `Boolean` (часть может отвалиться по транзиентному rate-limit
  API — это норма, не ошибка данных). Передавать агентам обрезку через `args`.
- Если в тексте данных нет — ставить `не_определено`, НЕ домысливать.

### Шаг 5. Агрегация
Считать доли с **правильными знаменателями**: для «доли марок» знаменатель — дела,
где марка названа (не вся выборка); «не определено» показывать явным процентом.
Где уместно — медиана и разброс (мин–макс, квартили), а не только среднее.

### Шаг 6. Справка (формат вывода)
Структура обязательная:

1. **Вопрос и параметры** — что искали, query/article/scope, период.
2. **Методика и ограничения** (НЕ опускать):
   - выборка N из total_found T; достижимый пул R; способ (random/stratified,
     разбивка по окнам); `seed` для воспроизводимости;
   - смещения: только опубликованные акты; пагинация sudact ≤500/запрос;
     обезличенные тексты; сортировка сайта.
   - формулировка: «Оценочная статистика по доступной выборке, не генеральная
     совокупность. Пригодна для ориентировки (вилка/тенденция), но НЕ как
     доказательство процессуальной статистики без оговорок.»
3. **Результаты** — таблицы с долями/медианами; под каждым ключевым числом —
   ссылки на 2–3 решения-примера из corpus.
4. **Выводы** — качественные, с оговорками.

## Замечания
- Токены: кодирование N полных текстов — основная стоимость. N=30 — недорогой
  пилот; N=100+ — лучше через Workflow.
- Воспроизводимость: фиксировать `--seed`; та же выборка повторяется.
- Этика/нагрузка: sudact ходит последовательно; не гнать огромные N подряд.
- Тот же скилл применим в Hermes (он на VPS, MCP подключается локально).
