# Анализ судебной практики (sudact) — два режима

Скилл поверх sudact MCP. Развилка по задаче пользователя:

- **Нужны цифры/доли/медианы по массиву дел** → **Режим A: Статистика**.
- **Нужно найти конкретные решения по трудному критерию** (редкий исход,
  конкретная позиция) → **Режим B: Адресный отлов**.

Бэкенды в репозитории `C:\Users\USER\Documents\Veritas\Code\sudact-mcp\`:
`stat_corpus.py` (выборка для A), `find_candidates.py` (кандидаты для B).
Запуск: `py -X utf8 <путь> ...` (пакет `mcp` уже стоит).

## Общие жёсткие принципы (оба режима)

1. **Никогда не выдавать ИИ-выводы как истину в последней инстанции.** Каждое
   утверждение — со ссылкой на конкретное решение sudact.
2. **Краткие выжимки ИИ-генерированы → подлежат сверке с первоисточником.**
   В проде агент уже один раз подставил чужое дело (галлюцинация). Всегда делать
   шаг QC и помечать выжимки как требующие проверки.
3. **Модель-кодировщик/проверяющий: Sonnet** (`agent(prompt,{model:'sonnet'})`).
   Калибровка 2026-06-26: Sonnet vs Opus 20/20 по критичным полям. Opus не нужен,
   Haiku для юр-задач рискован.
4. **Ограничения sudact:** ≤~500 результатов на запрос (пагинация зацикливается
   на 50-й стр.); только опубликованные, обезличенные акты; буквальный поиск
   (короткие запросы). Фильтр по датам с обоими пределами глючит на отдельных
   годах (детектор в `stat_corpus.py`).

# Режим A: Доказательная статистика

Повторяет метод А.Зазулина (sudact + парсер + LLM), но чинит его дыры (выдавал
«первые 100» за репрезентативную выборку). Доп. принципы:
- Результат — всегда *оценочная статистика по доступной выборке*, не генеральная.
- Не путать точность ИЗВЛЕЧЕНИЯ с РЕПРЕЗЕНТАТИВНОСТЬЮ.
- Честная выборка обязательна: `random` или `stratified` по датам, не «первые N».
- «Не определено» — отдельная категория, не выкидывать молча.

### A1. Уточнить вопросы + параметры (query/article/scope, период, размер N 30–50).
### A2. Разведка объёма: `total_found` (search или `stat_corpus.py --mode random --n 1`).
- total ≲ 500 → `--mode random`; total большой → `--mode stratified --windows ...`
  по годам, толстые годы (reachable=500<total) дробить на кварталы.
### A3. Собрать выборку с текстами:
```
py -X utf8 stat_corpus.py --query "<...>" [--article "<...>"] --scope regular \
   --mode stratified --windows "2022,2023,2024,2025" --n 50 --with-text \
   --seed 42 --out corpus.jsonl
```
Отчёт о методике (total/reachable/окна/picked) сохранить дословно — пойдёт в справку.
### A4. Кодирование: читать `text_excerpt` (обрезка, ~×11 дешевле полного текста).
Схема, напр.: `{doc_id,url,основной_состав,смежные_составы[],вид_наказания,срок,
реальное_или_условное,сумма|null,не_определено?}`. `основной_состав` при
совокупности = НАИБОЛЕЕ ТЯЖКАЯ статья, прочее в `смежные_составы`.
≤~40 — в контексте; больше — Workflow (Sonnet, `.catch(()=>null)`+`Boolean`).
### A5. Агрегация: правильные знаменатели (доля марок — от дел, где марка названа);
медиана и разброс, а не только среднее.
### A6. Справка: (1) вопрос+параметры; (2) **Методика и ограничения** —
N из total, способ выборки, seed, смещения, формулировка «оценочная, не
генеральная»; (3) результаты с долями/медианами и ссылками; (4) выводы с оговорками.

# Режим B: Адресный отлов (найти конкретные редкие решения)

Для случаев, где случайная выборка бесполезна (редкий исход <1–2%). Логика:
широкий **recall** разными запросами → строгая **проверка** каждого кандидата →
**QC** → справка. Проверено в бою (оправдания по ст.160: из 19 кандидатов — 10
подтверждённых, 9 отсеяно как «просьба»/«другая статья»).

### B1. Сформулировать «иголку» и углы поиска
- Чёткий критерий: что именно ищем (напр. «вынесен оправдательный приговор именно
  по ст.160», «взыскано > N», «суд применил позицию X»).
- **Несколько углов-запросов** под язык резолютива целевого исхода (поиск
  буквальный — одной формулировкой не выловить). Напр. для оправданий:
  `оправдательный приговор оправдать` || `оправдать отсутствие состава 302` ||
  `признать невиновным за отсутствием состава`.
- **Правильные разделы:** для редких исходов **ВС РФ (`vsrf`) часто богаче** —
  обжалование концентрирует там редкие исходы (прокуратура оспаривает оправдания).
  Плюс `regular` (первая инстанция), при необходимости `magistrate`.

### B2. Собрать кандидатов
```
py -X utf8 find_candidates.py \
   --queries "угол1||угол2||угол3" --article "160 УК РФ" \
   --scopes regular,vsrf --per 12 --out candidates.jsonl
```
Вывод дедуплицирован и отсортирован по `hit_count` (найден несколькими углами =
сильнее). Прочитать список (он короткий — только метаданные, без текстов).

### B3. Проверить каждого кандидата (Workflow, Sonnet)
Агент тянет ПОЛНЫЙ текст по URL через MCP и строго квалифицирует. Критично
различать: РЕАЛЬНЫЙ целевой исход vs лишь ПРОСЬБА стороны vs исход по ДРУГОЙ
статье vs процессуальное упоминание / прекращение (≠ оправдание). Default = «нет».
Скелет воркфлоу (адаптировать схему под критерий):
```js
const SCHEMA = { type:'object', additionalProperties:false, properties:{
  is_target:{type:'boolean', description:'true только если РЕАЛЬНО искомый исход именно по нужной статье (полный/частичный)'},
  is_request_only:{type:'boolean', description:'true если это лишь просьба/жалоба, а суд не вынес искомое'},
  scope_of_match:{type:'string', enum:['полное','частичное_по_эпизодам','по_другой_статье','нет']},
  instance:{type:'string', enum:['первая','апелляция','кассация','надзор','иное']},
  final_status:{type:'string', enum:['вынесено','оставлено_в_силе','отменено','изменено','не_применимо']},
  article_part:{type:'string'}, basis:{type:'string'},
  summary:{type:'string', description:'1-2 предложения по фактам ИЗ ТЕКСТА'},
  confidence:{type:'string', enum:['высокая','средняя','низкая']},
}, required:['is_target','is_request_only','scope_of_match','instance','final_status','article_part','basis','summary','confidence'] }

const items = typeof args==='string' ? JSON.parse(args) : args
const coded = await parallel(items.map(it => async () => {
  try { return { ...it, ...await agent(prompt(it.url), {model:'sonnet', schema:SCHEMA, phase:'Проверка', label:`#${it.i}`}) } }
  catch { return null }
}))
const ok = coded.filter(Boolean)
return { confirmed: ok.filter(c=>c.is_target), requests: ok.filter(c=>!c.is_target&&c.is_request_only), other: ok.filter(c=>!c.is_target&&!c.is_request_only) }
```
Промпт агенту: «ШАГ 1 — получи полный текст через MCP `get_court_decision`
(url_or_id=URL; при отсутствии — ToolSearch `select:mcp__sudact__get_court_decision`).
ШАГ 2 — квалифицируй СТРОГО по схеме, только по тексту, различая реальный исход /
просьбу / другую статью / прекращение.»

### B4. QC — ОБЯЗАТЕЛЬНО (не пропускать)
- **Ищи дубли:** одинаковые/почти одинаковые `summary` у разных дел = агент
  подставил чужое (так было с делом №3). Любой такой кандидат — перечитать самому
  через `get_court_decision` и исправить/выкинуть.
- Спот-чек 1–2 пограничных подтверждённых против сниппета/первоисточника.
- Если текст огромный (>~250k симв.) — `get_court_decision` сохранит его в файл;
  грепни резолютив (`ПРИГОВОРИЛ`/`оправдать`/`ст.160`) или слайсь питоном.

### B5. Справка
- **Подтверждённые** сгруппировать логично (инстанция; исход; устояло/отменено),
  таблица со ссылками + отличающий критерий (часть статьи, основание, статус).
- **Отсеянные** — числом и причиной (просьба / другая статья / прекращение).
- **Дисклеймер:** «Не исчерпывающий список — проверенные “иголки” из верхушки
  выдачи sudact; для полноты нужен платный корпус с фильтром по исходу. Выжимки
  ИИ-генерированы — перед использованием открыть сам акт.»

## Замечания (оба режима)
- Токены: основная стоимость — чтение полных текстов агентами. В A кормить
  обрезку; в B полный текст оправдан (нужна резолютивная точность), но держать
  пул кандидатов разумным (≤~25–30 на проверку).
- Воспроизводимость A: фиксировать `--seed`.
- Workflow жжёт токены и требует явного согласия пользователя — предупреждать.
- Этика/нагрузка: sudact ходит последовательно; не гнать огромные объёмы подряд.
- Тот же скилл применим в Hermes (он на VPS, MCP подключается локально).
